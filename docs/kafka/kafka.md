# Kafka 개념 정리 문서

## 1. 이벤트 아이디어의 시스템 확장 관점

### 1.1 기존 구조: 내부 이벤트 방식

기존 Application Event 방식은 **하나의 애플리케이션 내부**에서만 이벤트를 전달한다.

```text
Service A
 └─ Event Publish → Event Listener (같은 프로세스)
```

특징:

* 같은 JVM 안에서만 동작
* 다른 서비스와 공유 불가
* 확장성 한계

### 1.2 확장 구조: Kafka 기반 이벤트 방식

Kafka를 도입하면 이벤트를 **시스템 전체로 확장**할 수 있다.

```text
Service A (Producer)
   ↓
Kafka Cluster
   ↓
Service B / C / D (Consumer)
```

의미:

* 이벤트를 시스템 자산으로 관리
* 여러 서비스가 동시에 활용 가능
* 마이크로서비스 아키텍처에 적합

### 1.3 핵심 개념: 이벤트 중심 아키텍처

Kafka는 다음 개념을 가능하게 한다.

* 이벤트를 "데이터"로 취급
* 서비스 간 직접 호출 제거
* 느슨한 결합(Loose Coupling)

즉,

> "기능 호출" → "이벤트 발행" 중심 구조로 전환

---

## 2. Kafka란 무엇인가?

Kafka는 **분산 이벤트 스트리밍 플랫폼**이다.

주요 역할:

* 메시지 저장
* 메시지 전달
* 메시지 재처리
* 대용량 데이터 처리

한 문장 요약:

> Kafka는 대규모 분산 환경에서 신뢰성 있게 이벤트를 저장·전달하는 시스템이다.

---

## 3. Kafka 시스템 구성 요소

### 3.1 Broker

* Kafka 서버 1대
* 메시지 저장소 역할
* 여러 대로 클러스터 구성

역할:

* 메시지 저장
* 복제 관리
* 요청 처리

---

### 3.2 Topic

* 메시지 분류 단위
* 하나의 이벤트 채널

예시:

```text
concert.reservation.completed
user.signup
order.paid
```

특징:

* 하나의 Topic에 여러 Consumer 연결 가능
* 이벤트 종류별로 분리 관리

---

### 3.3 Partition

* Topic의 물리적 분할 단위
* 병렬 처리 핵심 요소

구조:

```text
Topic
 ├─ Partition 0
 ├─ Partition 1
 └─ Partition 2
```

특징:

* 같은 Key → 같은 Partition
* Partition 수 = 최대 병렬 처리 수

---

### 3.4 Offset

* Partition 내부 메시지 번호
* 처리 위치 표시자

예시:

```text
Partition 0
0 → 1 → 2 → 3 → 4
```

특징:

* Consumer가 직접 관리
* 재시작 시 재처리 가능

---

### 3.5 Consumer Group

* Consumer들의 논리적 묶음

구조:

```text
Group A
 ├─ Consumer 1 → Partition 0
 └─ Consumer 2 → Partition 1
```

특징:

* 그룹 내 중복 처리 없음
* 그룹 간 독립 처리

## 4. Kafka 제공 기능 정리

### 4.1 메시지 영속성

* 디스크에 저장

* Offset 조작 가능
* 과거 데이터 재처리 가능

### 4.3 고가용성

* Replication 지원
* Broker 장애 대응

### 4.4 고성능 처리

* 초당 수십만 ~ 수백만 메시지 처리
* 순차 디스크 I/O 활용

### 4.5 확장성

* Broker/Partition 추가 가능
* 수평 확장 구조

## 6. Kafka 단점 (매우 중요)

### 6.1 운영 복잡도

Kafka는 설정과 운영이 어렵다.

문제점:

* 클러스터 관리
* 모니터링 필요
* 장애 대응 복잡

→ 소규모 시스템에는 과도할 수 있음

---

### 6.2 개발 난이도 증가

고려 요소 증가:

* 중복 처리
* 순서 보장
* 재시도
* 에러 처리
* 멱등성

→ 단순 CRUD보다 난이도 높음

---

### 6.3 데이터 일관성 문제

Kafka는 기본적으로 **Eventually Consistent** 구조이다.

문제:

* 즉시 반영 보장 어려움
* 실시간 트랜잭션과 충돌 가능

---

### 6.4 정확히 한 번 처리의 어려움

Exactly-once 구현은 복잡하다.

필요 요소:

* Transactional Producer
* Idempotent Consumer
* Outbox 패턴

→ 구현 비용 큼

---

### 6.5 리소스 비용

* 디스크 사용량 큼
* 메모리/네트워크 사용 증가
* 서버 비용 상승

---

## 7. MQ / Event / Kafka 비교

| 구분  | Application Event | MQ    | Kafka  |
| --- | ----------------- | ----- | ------ |
| 범위  | 프로세스 내부           | 서비스 간 | 시스템 전체 |
| 저장  | 메모리               | 제한적   | 디스크    |
| 확장성 | 낮음                | 중간    | 매우 높음  |
| 재처리 | 불가                | 제한    | 가능     |
| 복잡도 | 낮음                | 중간    | 높음     |

## 8. 서비스 적용 구조 예시

### 8.1 기본 흐름

```text
Controller
  ↓
Service (DB 처리)
  ↓
Kafka Producer
  ↓
Kafka Cluster
  ↓
Kafka Consumer
  ↓
외부 시스템 연동
```

---

### 8.2 트랜잭션 흐름

```text
DB Commit
   ↓
Kafka Publish
   ↓
Consumer 처리
```

주의:

* DB와 Kafka 불일치 가능
* Outbox 패턴 필요

## 9. Kafka 기반 시스템의 특징

### 9.1 아키텍처 특징


## 11. Kafka 도입 시 판단 기준

### 11.1 도입이 적합한 경우

* 서비스가 여러 개
* 이벤트 재활용 필요
* 대용량 처리 필요
* 분석 시스템 연동 필요

### 11.2 도입이 부적합한 경우

* 단일 서버
* 소규모 서비스
* 단순 CRUD
* 실시간 동기 트랜잭션 위주
