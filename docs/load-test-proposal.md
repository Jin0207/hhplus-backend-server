# 부하테스트 계획서

> 작성일: 2026-02-26
> 대상 시스템: HHPlus 이커머스 백엔드 서버
> 테스트 범위: 선착순 쿠폰 발급 API

---

## 1. 왜 지금 부하테스트가 필요한가

### 1-1. 비즈니스 리스크

선착순 쿠폰 발급은 **짧은 시간에 폭발적인 트래픽이 집중**되는 이벤트입니다.

- 쿠폰 수량은 유한합니다 (예: 100장).
- 수천 명의 사용자가 동시에 발급 버튼을 누릅니다.
- 이 순간 시스템이 버티지 못하면 **서비스 장애 = 매출 손실 + 브랜드 신뢰 하락**으로 직결됩니다.

이미 코드에는 Redis + DB 이중 동시성 제어가 구현되어 있습니다.
**하지만 "코드가 맞다"와 "실제 부하에서 버틴다"는 전혀 다른 이야기입니다.**

### 1-2. 현재 시스템의 동시성 구조

```
사용자 요청
  │
  ▼
[Redis Set - SADD]         ← 중복 발급 원천 차단 (Atomic)
  │
  ▼
[Redis Sorted Set - ZADD]  ← 선착순 대기열 등록 (timestamp score)
  │
  ▼
[Redis ZRANK 검증]          ← 수량 초과 여부 판단
  │
  ▼
[MySQL SELECT FOR UPDATE]  ← 비관적 잠금으로 수량 감소 (DB 정합성 보장)
  │
  ▼
UserCoupon 생성 + Coupon.availableQuantity 차감
```

이 구조는 **정상 동작**은 보장하지만, **부하 상황에서의 성능 특성**은 아직 검증되지 않았습니다.

> Redis → DB 순서의 이중 잠금 구조는
> DB 병목이 심해질수록 Redis에 대기 큐가 쌓이고,
> 타임아웃 → 롤백 → 재시도 폭풍으로 이어질 수 있습니다.

## 2. 무엇을 테스트할 것인가

### 핵심 API: `POST /api/v1/coupons/issue`

| 항목 | 내용 |
|------|------|
| 엔드포인트 | `POST /api/v1/coupons/issue` |
| 인증 | JWT Bearer Token |
| 요청 바디 | `{ "couponId": 1 }` |
| 동시성 제어 | Redis Set + Sorted Set + MySQL Pessimistic Lock |
| 기대 결과 | 수량만큼만 성공, 나머지는 정상적으로 실패 (overselling 없음) |

### 검증 항목

| 검증 포인트 | 정상 기준 | 리스크 상황 |
|------------|----------|------------|
| **처리량 (Throughput)** | 목표 TPS 달성 여부 | TPS 미달 → 대기열 폭증 |
| **응답 시간 (Latency)** | P95 < 500ms | P95 > 2s → 사용자 이탈 |
| **정확성 (Accuracy)** | 발급 수량 = 쿠폰 수량 | Overselling → 재정 손실 |
| **중복 방지 (Dedup)** | 동일 유저 1회만 성공 | 중복 발급 → 비용 손실 |
| **오류율 (Error Rate)** | < 1% (정상 실패 제외) | > 5% → 시스템 불안정 |
| **DB 커넥션 고갈** | 커넥션 풀 여유 유지 | 고갈 → 전체 서비스 영향 |

## 3. 어떻게 테스트할 것인가

### 3-1. 도구

| 도구 | 용도 |
|------|------|
| **k6** (권장) | 시나리오 기반 부하 생성, VU(가상 사용자) 제어 |
| **JMeter** | 기존 `coupon_concurrency_test.jmx` 활용 가능 |
| **Grafana + InfluxDB** | 실시간 메트릭 시각화 |
| **Spring Actuator + Prometheus** | 서버 내부 메트릭 (JVM, DB 커넥션 풀, Redis 커맨드 수) |

> 기존 JMeter 파일(`coupon_concurrency_test.jmx`)이 이미 존재합니다.
> 100명 동시 요청 시나리오가 구성되어 있으므로, 이를 베이스라인으로 활용합니다.

### 3-2. 테스트 시나리오

#### Scenario 1 - 정상 부하 (Smoke Test)
```
목적: 기본 동작 확인
VU: 50명 동시
쿠폰 수량: 50장
기대: 정확히 50건 성공, 0건 중복
```

#### Scenario 2 - 경쟁 부하 (Load Test) ← **핵심**
```
목적: 실제 이벤트 상황 시뮬레이션
VU: 500명 동시 (램프업 30초)
쿠폰 수량: 100장
기대: 정확히 100건 성공, P95 응답 < 500ms
```

#### Scenario 3 - 극한 부하 (Stress Test)
```
목적: 시스템 한계치 탐색
VU: 1000명 → 2000명 → 3000명 (단계별 증가)
쿠폰 수량: 100장
기대: Overselling 없음, 한계 TPS 확인, 오류율 추이
```

#### Scenario 4 - 중복 요청 (Idempotency Test)
```
목적: Redis Set(SADD) 중복 차단 검증
VU: 1명이 100번 연속 요청
기대: 1건만 성공, 99건은 COUPON_ALREADY_ISSUED(400)
```

### 3-3. 테스트 환경

```
[ k6 / JMeter ]
       │
       ▼ HTTP
[ Spring Boot App ]  ← 로컬 or 스테이징
       │
       ├── [ Redis ]    (localhost:6379)
       └── [ MySQL ]    (localhost:3306, docker-compose)
```

- 로컬 환경: `docker-compose up`으로 Redis + MySQL 기동
- 테스트 전: 쿠폰 데이터 INSERT, Redis 키 초기화 (DEL `coupon:*`)
- 테스트 후: 발급 건수 집계 쿼리로 정확성 검증

```sql
-- 발급 건수 검증
SELECT COUNT(*) FROM user_coupon WHERE coupon_id = 1;

-- 중복 발급 검증
SELECT user_id, COUNT(*) cnt
FROM user_coupon
WHERE coupon_id = 1
GROUP BY user_id
HAVING cnt > 1;
```

## 4. 기대 결과 및 판단 기준

### Pass/Fail 기준

| 지표 | Pass | Fail |
|------|------|------|
| 발급 수량 정확성 | 쿠폰 수량과 일치 | 초과 또는 미달 |
| 중복 발급 | 0건 | 1건 이상 |
| P95 응답시간 (부하 시) | < 500ms | > 2,000ms |
| 오류율 (비즈니스 오류 제외) | < 1% | > 5% |
| 서버 정상 복구 | 부하 종료 후 정상 응답 | 재기동 필요 |

### 성능 이슈 발견 시 개선 방향

| 병목 원인 | 개선 방안 |
|----------|----------|
| MySQL 커넥션 대기 | HikariCP 풀 사이즈 조정, 트랜잭션 범위 최소화 |
| Redis 직렬 처리 | Lua Script 활용으로 Redis 왕복 횟수 최소화 |
| 쿠폰 발급 DB Write 집중 | 비동기 처리 전환 (Kafka 기반 쿠폰 발급 큐) |
| 전체 응답 지연 | 수량 체크를 Redis에서만 처리하고 DB는 배치로 후기록 |

## 8. 개선 방향 및 권고사항

### 즉시 적용 가능한 개선

**① HikariCP 풀 사이즈 조정** ✅ 적용 완료 (`maximum-pool-size: 20`)

**② 서버 Thread Pool 조정** (다음 단계)

```yaml
server:
  tomcat:
    threads:
      max: 400       # 500 VU 대응 (기본 200 → 증가)
      min-spare: 20
    accept-count: 200  # 큐 대기 허용
```

### 중장기 개선

| 우선순위 | 개선 방안 | 효과 |
|---------|----------|------|
| High | Tomcat `threads.max` 400으로 증가 | 500 VU 서버 크래시 방지 |
| High | Redis에서 수량 차감 (Lua Script) 후 DB는 비동기 후기록 | DB 락 경합 제거, 응답시간 단축 |
| Medium | 쿠폰 발급을 Kafka 큐로 전환 (비동기) | DB 트랜잭션 분리, 안정적 처리량 확보 |
| Low | Grafana + Prometheus 연동으로 실시간 모니터링 구성 | 이벤트 당일 병목 즉시 감지 |

## 9. 결론

> **"쿠폰 이벤트 당일, 500명이 동시에 누를 때 우리 서버는 버티는가?"**

### 1차 결과 (pool-size=3): 버티지 못함
- 50명 동시: 32건만 성공, 나머지 DB 커넥션 타임아웃
